# 第一届 CCF 车联网安全攻防赛线上赛题目剖析

## 一、赛事与赛题概述

第一届 CCF 车联网安全攻防赛线上赛采用典型的 Jeopardy 解题模式，参赛队伍通过分析附件、连接远程服务、寻找程序漏洞或还原密码算法获取 Flag，并按照题目分值累计比赛成绩。

从公开赛题复盘来看，本次线上赛共涉及 Crypto、Web、IOV、Misc、Reverse 和 PWN 六个方向，主要题目包括：

| 题目方向 | 题目名称 |
|---|---|
| Crypto | Curve、nfsr |
| Web | Easylogin、Serp |
| IOV | VIN |
| Misc | GPS、OTA-流量 |
| Reverse | ezapk、Algorithm |
| PWN | pwn-3 |

整体来看，本次比赛并不是完全围绕 CAN 总线、UDS 诊断和 ECU 固件开展的纯汽车协议攻防，而是以传统 CTF 技术为基础，将 VIN、GPS、蓝牙通信、车辆访问凭证和 OTA 升级等汽车业务元素融入题目。

因此，本次线上赛可以概括为：

> 以传统 CTF 安全能力为基础，以智能汽车业务数据和车联网应用场景为载体的综合性汽车安全赛事。

赛题既考查密码学、Web 安全、逆向工程、二进制漏洞利用等传统能力，也考查参赛选手对车辆身份、移动端应用、蓝牙通信、定位轨迹和 OTA 升级流程的理解。

## 二、Crypto 方向赛题剖析

### 2.1 Curve

#### 1. 题目原理

Curve 是一道曲线密码学题目。题目构造了一个类似密钥交换的密码系统，通信双方分别生成私钥和公钥，并利用双方的公私钥计算共享秘密。

程序随后对共享秘密的横坐标进行 SHA-256 运算，将其结果作为 AES 密钥，对 Flag 进行加密。

其基本过程可以表示为：

```text
生成曲线参数
    ↓
生成通信双方私钥
    ↓
计算对应公钥
    ↓
计算共享秘密
    ↓
SHA-256 派生 AES 密钥
    ↓
AES 加密 Flag
```

选手需要根据题目给出的曲线点和运算结果，恢复隐藏的曲线参数或私钥，最终重新计算共享秘密并解密 Flag。

#### 2. 主要解题思路

该题的关键在于分析题目使用的曲线结构，并找到曲线群与有限域乘法群之间的映射关系。

主要步骤包括：

1. 根据已知曲线点满足的等式构造多个整数关系；
2. 通过最大公约数恢复有限域模数；
3. 将曲线上的点运算转化到有限域乘法群中；
4. 将曲线离散对数问题转化为普通有限域离散对数问题；
5. 使用 SageMath 中的 `discrete_log` 求解私钥；
6. 根据私钥重新计算共享秘密；
7. 对共享秘密进行 SHA-256；
8. 使用派生出的 AES 密钥解密密文。

#### 3. 考查知识点

该题主要考查：

- 有限域理论；
- 群论基础；
- 曲线密码学；
- 离散对数问题；
- 群同态和结构映射；
- AES 对称加密；
- SHA-256 密钥派生；
- SageMath 数学工具使用。

#### 4. 汽车安全特征

Curve 题目本身与汽车业务的直接联系较弱，其核心仍然是传统 Crypto 题目。

在实际车联网系统中，曲线密码学可能用于：

- 车辆与云端身份认证；
- 车载终端密钥协商；
- OTA 升级包签名；
- V2X 通信认证；
- 数字证书和安全通信。

但本题并没有将密码算法与具体车载协议、ECU 或 OTA 签名机制结合，因此更适合作为参赛选手密码学基础能力的考查题。

### 2.2 nfsr

#### 1. 题目原理

nfsr 是一道基于移位寄存器的流密码分析题。

题目使用多组 128 位移位寄存器，根据不同反馈掩码进行状态更新，并通过非线性组合函数生成输出序列。同时，原始寄存器状态还会被作为 AES 密钥，用于加密 Flag。

题目的核心结构为：

```text
初始寄存器状态
    ↓
多组移位寄存器迭代
    ↓
非线性函数组合输出
    ↓
生成已知密钥流
```

与此同时：

```text
初始状态
    ↓
转换为 AES 密钥
    ↓
加密 Flag
```

选手需要根据题目提供的输出序列，反推出寄存器初始状态。

#### 2. 主要解题思路

由于移位寄存器的状态更新过程可以表示为 \(GF(2)\) 上的线性变换，因此可以将整个寄存器迭代过程转换为矩阵运算。

主要步骤包括：

1. 将每个寄存器状态表示为二进制向量；
2. 将反馈和移位操作表示为状态转移矩阵；
3. 根据已知输出构造关于初始状态的线性方程；
4. 在 \(GF(2)\) 上求解线性方程组；
5. 分析方程组的右核，确定可能存在的自由变量；
6. 枚举满足输出条件的候选初始状态；
7. 将候选状态转换为 AES 密钥；
8. 尝试解密密文，并通过填充或 Flag 格式判断正确结果。

#### 3. 考查知识点

该题主要考查：

- LFSR 和 NFSR；
- 流密码原理；
- \(GF(2)\) 上的线性代数；
- 状态转移矩阵；
- 已知密钥流攻击；
- AES 解密；
- SageMath 矩阵运算。

#### 4. 汽车安全特征

nfsr 题目的直接汽车属性同样较弱。

不过在真实车辆安全中，移位寄存器和轻量密码算法可能应用于：

- 遥控钥匙；
- 滚动码系统；
- 无钥匙进入系统；
- TPMS 胎压监测通信；
- 低功耗无线设备；
- 车载传感器认证。

如果赛题进一步结合遥控钥匙抓包数据、滚动码同步机制或车辆解锁认证流程，其汽车安全特征会更加明显。

## 三、Web 方向赛题剖析

### 3.1 Easylogin

#### 1. 题目原理

Easylogin 是一道典型的身份认证 Token 伪造题，核心漏洞为哈希长度扩展攻击。

系统生成 Token 时，可能采用如下结构：

```text
Hash(secret || message)
```

其中：

- `secret` 为服务端保存的密钥；
- `message` 为用户身份、权限或时间戳等信息；
- `Hash` 为具有 Merkle-Damgård 结构的哈希算法。

当服务端仅通过重新计算哈希验证消息完整性时，攻击者即使不知道 `secret`，也可能在原始消息后追加新的权限字段，并计算出合法的新哈希值。

#### 2. 主要解题思路

主要流程为：

1. 注册或登录普通用户账号；
2. 获取系统返回的原始 Token；
3. 对 Token 进行 Base64、URL 编码或字段结构分析；
4. 判断 Token 中是否存在消息与哈希拼接结构；
5. 推测服务端密钥长度；
6. 使用哈希长度扩展工具追加管理员权限字段；
7. 生成新的消息和摘要；
8. 按照系统格式重新进行 URL 编码或 Base64 编码；
9. 使用伪造 Token 访问管理员功能；
10. 获取 Flag。

文章分析中根据 PHP `uniqid()` 的特点推测密钥长度为 13 字节，从而缩小了密钥长度枚举范围。

#### 3. 考查知识点

该题主要考查：

- Web Token 结构分析；
- MD5、SHA-1 等哈希算法原理；
- Merkle-Damgård 结构；
- 哈希长度扩展攻击；
- Base64 编解码；
- URL 编码；
- Web 权限绕过。

#### 4. 汽车安全特征

该题在技术上属于标准 Web CTF。

如果将其放到车联网环境中，可能对应：

- 车辆远程控制接口 Token；
- 车主 App 身份凭证；
- 车辆管理平台权限认证；
- 车队管理后台；
- OTA 管理员接口。

但公开题目中并没有体现具体车辆业务，因此汽车属性主要体现在比赛场景，而不是漏洞本身。

### 3.2 Serp

#### 1. 题目原理

Serp 是一道 PHP 代码审计和反序列化利用题。

题目的完整攻击链可以概括为：

```text
目录扫描
    ↓
发现 .git 目录泄露
    ↓
恢复网站源代码
    ↓
审计 PHP 类和魔术方法
    ↓
构造反序列化对象链
    ↓
利用文件读取能力
    ↓
读取目标 PHP 文件
    ↓
获取 Flag
```

#### 2. 主要解题思路

首先通过目录扫描发现网站存在 `.git` 目录泄露，随后使用 GitHack 等工具恢复项目源码。

得到源码后，重点审计：

- `unserialize()` 调用位置；
- 用户输入是否可控；
- `__construct()`；
- `__destruct()`；
- `__toString()`；
- `__get()`；
- `__call()`；
- 动态实例化类；
- 动态访问属性或方法。

题目中可以通过对象属性控制动态实例化的类，并将其设置为 `SplFileObject`，再配合 PHP 伪协议读取目标文件。

典型载荷形式为：

```php
php://filter/convert.base64-encode/resource=f1a9.php
```

程序读取文件后返回 Base64 编码内容，选手解码即可得到源码及 Flag。

#### 3. 考查知识点

该题主要考查：

- 目录和敏感文件扫描；
- `.git` 源码泄露；
- Git 仓库恢复；
- PHP 源码审计；
- PHP 反序列化；
- 魔术方法；
- POP 链构造；
- `SplFileObject`；
- `php://filter`；
- 任意文件读取。

#### 4. 汽车安全特征

该题与车辆专有技术联系较少，本质上属于传统 Web 漏洞利用。

不过现实中的车联网云平台、经销商管理系统、车主服务平台和 OTA 后台同样可能使用 PHP、Java 或其他 Web 技术。一旦存在源码泄露、反序列化或文件读取漏洞，攻击者可能进一步获取：

- 数据库账号；
- 云平台接口密钥；
- 车辆控制接口；
- OTA 签名密钥；
- 用户隐私数据；
- VIN 与车主绑定关系。

因此传统 Web 安全能力仍然是车联网安全体系中的重要组成部分。

## 四、IOV 方向赛题剖析

### 4.1 VIN

VIN 是本次线上赛中汽车安全特色较为明显的一道综合题。

#### 1. 题目背景

VIN 是车辆识别代码，用于唯一标识车辆身份。题目提供了车辆蓝牙通信流量和 Android 应用程序，要求选手分析通信数据并恢复被加密的 VIN 或相关 Flag 信息。

#### 2. 题目攻击链

完整解题链路为：

```text
分析蓝牙通信流量
    ↓
定位成功认证或数据传输报文
    ↓
提取十六进制数据
    ↓
反编译 Android APK
    ↓
定位 VIN 处理函数
    ↓
追踪 JNI Native 方法
    ↓
使用 IDA 分析 SO 文件
    ↓
恢复硬编码密钥和加密算法
    ↓
编写逆向解密脚本
    ↓
恢复 VIN 或 Flag
```

#### 3. 蓝牙流量分析

选手首先需要分析题目提供的通信流量，寻找具有明显规律的数据包。

重点关注：

- 蓝牙写请求；
- 蓝牙通知报文；
- 认证成功前后的数据变化；
- 长度固定的十六进制数据；
- 多次编码或加密后的字符串。

通过对流量进行对比，可以找到疑似 VIN 密文的数据。

#### 4. APK 逆向分析

使用 JADX 反编译 Android APK 后，可以搜索：

- VIN；
- encrypt；
- decrypt；
- verify；
- key；
- native；
- JNI；
- Bluetooth。

Java 层通常负责：

- 接收用户输入；
- 读取车辆数据；
- 调用 Native 方法；
- 将处理结果通过蓝牙发送。

题目将关键加密逻辑放在 Native SO 文件中，仅分析 Java 层无法直接得到完整算法。

#### 5. JNI 和 SO 逆向

选手需要提取 APK 中的 `.so` 文件，并使用 IDA 等工具进行逆向分析。

重点分析：

- `JNI_OnLoad`；
- `Java_包名_类名_方法名`；
- 动态注册的 JNI 方法；
- 字符串常量；
- 硬编码密钥；
- 循环异或；
- 字节交换；
- 移位和加减运算。

题目中可以发现硬编码密钥：

```text
MySecretKey123!@#
```

进一步还原 Native 层自定义字节处理算法后，即可编写相反的解密逻辑。

#### 6. 考查知识点

该题综合考查：

- 蓝牙通信流量分析；
- 十六进制数据识别；
- Android APK 逆向；
- JADX 使用；
- JNI 调用追踪；
- Native SO 逆向；
- IDA 静态分析；
- ARM 汇编基础；
- 硬编码密钥发现；
- 自定义加密算法还原；
- Python 解密脚本编写。

#### 7. 题目特点

VIN 题目的单个技术点并不陌生，但它将多个传统 CTF 能力串联成一条完整车联网分析链路：

```text
无线通信
+ 移动应用
+ Native 程序
+ 密码算法
+ 车辆身份数据
```

这种跨层题目比单纯 APK 逆向更接近真实车联网安全测试。

实际车辆测试中经常需要：

1. 抓取手机 App 与车辆之间的蓝牙通信；
2. 逆向手机 App；
3. 找到通信协议封装逻辑；
4. 分析 JNI 和 SO；
5. 恢复指令格式或密钥；
6. 构造新的车辆控制报文。

不过该题主要围绕数据恢复展开，并没有进一步涉及蓝牙配对绕过、重放攻击、车辆控制或蓝牙协议栈漏洞。

## 五、Misc 方向赛题剖析

### 5.1 GPS

#### 1. 题目原理

GPS 是一道车辆轨迹数据分析题。题目提供 GPX 格式的轨迹文件，选手需要从大量正常轨迹点中找到异常位置，并确定异常点对应的时间。

GPX 文件通常采用 XML 结构，轨迹点可能包含：

```xml
<trkpt lat="纬度" lon="经度">
    <ele>海拔</ele>
    <time>时间</time>
</trkpt>
```

#### 2. 主要解题思路

主要步骤包括：

1. 使用文本编辑器、脚本或地图工具打开 GPX 文件；
2. 提取所有经纬度坐标；
3. 将轨迹点绘制到地图；
4. 观察轨迹中是否存在明显偏离正常路径的位置；
5. 确定异常经纬度坐标；
6. 在 GPX 原文件中搜索该坐标；
7. 获取对应时间戳；
8. 按照题目要求提交时间或 Flag。

#### 3. 考查知识点

该题主要考查：

- GPX 文件格式；
- XML 数据解析；
- GPS 经纬度；
- 轨迹可视化；
- 时间戳分析；
- 空间异常检测；
- 车辆轨迹取证。

#### 4. 题目特点

传统 Misc 题可能要求分析系统日志、网络日志或用户操作记录，而 GPS 题将分析对象替换为车辆轨迹数据。

其汽车安全属性主要体现在数据来源：

- 车辆导航轨迹；
- 车载定位数据；
- 行车路径记录；
- 时间和空间关联分析。

但该题没有进一步涉及：

- GPS 欺骗；
- GNSS 信号干扰；
- 虚假坐标注入；
- 车载导航劫持；
- 定位融合算法攻击。

因此，这道题更接近车辆数据取证，而不是 GPS 信号层攻击。

### 5.2 OTA-流量

OTA-流量是本次线上赛中车联网业务特征最明显的一道题。

#### 1. 题目背景

OTA 即远程在线升级。现代智能汽车通常通过 T-Box 或车机连接云端服务器，获取软件、配置文件或 ECU 固件升级包。

题目提供 OTA 通信流量，选手需要分析车辆与服务器之间的握手过程，获取会话密钥，并解密加密传输的升级文件。

#### 2. 题目数据组成

流量中包含多个重要文件或数据阶段：

- `handshake`：车辆与 OTA 服务器握手数据；
- `init`：OTA 初始化数据；
- `download`：升级文件或下载数据。

其中 `handshake` 为 JSON 格式，可能包含：

- VIN；
- 时间戳；
- 电量；
- 里程；
- 车型；
- `session_key`；
- `session_token`；
- 车辆权限信息。

#### 3. 主要解题思路

完整分析过程可以表示为：

```text
打开 PCAP 流量
    ↓
识别 OTA 相关会话
    ↓
导出 handshake 数据
    ↓
解析 JSON 字段
    ↓
对比不同 VIN 的服务器响应
    ↓
发现特殊车辆身份
    ↓
提取 session_key
    ↓
导出 init 和 download 文件
    ↓
判断 AES 工作模式
    ↓
解密 OTA 数据
    ↓
获取 Flag
```

题目中的特殊 VIN 为：

```text
VINADMIN12345VSEC
```

当该 VIN 与服务器握手时，服务端会返回特殊响应，例如：

```json
{
    "admin_flag": true,
    "hint": "Special vehicle detected",
    "vehicle_model": "ModelAdmin"
}
```

说明服务端根据 VIN 判断车辆权限或升级类型。

随后从握手响应中获取 `session_key`，使用 AES-CBC 对 OTA 数据进行解密。题目中使用全零 IV，因此解密参数可以表示为：

```text
Algorithm：AES
Mode：CBC
Key：handshake 中的 session_key
IV：16 字节全零
```

#### 4. 考查知识点

该题主要考查：

- Wireshark 流量分析；
- PCAP 会话筛选；
- HTTP 或自定义协议分析；
- 文件导出；
- JSON 数据解析；
- VIN 身份识别；
- 会话关联；
- AES-CBC；
- 会话密钥提取；
- OTA 数据解密；
- Python 密码脚本编写。

#### 5. 汽车安全特征

与普通流量分析题相比，OTA-流量题增加了完整的车辆升级业务语义。

传统流量题通常为：

```text
分析 HTTP/FTP/SMB 流量
    ↓
导出文件
    ↓
解码或解密
    ↓
获得 Flag
```

OTA 题则变为：

```text
车辆发起升级请求
    ↓
上报 VIN 和车辆状态
    ↓
服务器识别车辆身份
    ↓
生成 OTA 会话密钥
    ↓
下发加密升级文件
    ↓
解密升级数据
```

选手不仅需要找到密钥，还必须理解握手数据、车辆身份、升级文件和会话之间的对应关系。

#### 6. 与真实 OTA 安全的差距

真实汽车 OTA 系统通常还涉及：

- TLS 双向认证；
- 车辆数字证书；
- 固件数字签名；
- 证书链验证；
- Secure Boot；
- 防回滚机制；
- 差分升级；
- A/B 分区；
- ECU 固件适配；
- UDS 刷写；
- 升级失败恢复；
- OTA 管理平台权限；
- 密钥安全存储。

本题主要考查从通信流量中提取会话密钥并解密文件，还没有深入到固件签名绕过、恶意升级包构造或 ECU 刷写。

## 六、Reverse 方向赛题剖析

### 6.1 ezapk

#### 1. 题目原理

ezapk 是一道 Android Native 逆向题。题目以车辆访问凭证为背景，将核心加密算法放置在 Native SO 文件中。

程序处理过程大致为：

```text
输入车辆访问数据
    ↓
Java 层调用 JNI
    ↓
Native 层执行魔改 RC4
    ↓
结果执行 RLE 编码
    ↓
生成最终凭证数据
```

#### 2. 主要解题思路

首先使用 JADX 分析 APK Java 代码，定位：

- 用户输入位置；
- Native 库加载；
- JNI 方法声明；
- 输入输出校验逻辑。

随后提取 SO 文件并使用 IDA 分析，发现算法主体类似 RC4，但存在以下修改：

- 使用自定义初始 S 盒；
- 使用固定 6 字节密钥；
- 对标准 RC4 的初始化或输出过程进行了调整；
- 加密结果还经过 RLE 编码。

解题时需要：

1. 还原魔改 RC4 初始化过程；
2. 还原密钥调度算法；
3. 还原伪随机序列生成过程；
4. 对目标数据进行 RC4 解密；
5. 对结果执行反向 RLE 解码；
6. 获得车辆访问凭证或 Flag。

#### 3. 考查知识点

该题主要考查：

- Android APK 逆向；
- JADX；
- JNI；
- Native SO；
- IDA；
- ARM 汇编；
- RC4；
- 自定义 S 盒；
- 密钥调度算法；
- RLE 编解码；
- 算法还原。

#### 4. 汽车安全特征

ezapk 使用了车辆访问凭证作为题目背景，但技术本质仍是传统 Android Native 逆向。

如果将车辆访问凭证替换为门禁凭证、设备注册码或普通 App Token，解题方式基本不会发生变化。

其主要价值在于提醒参赛选手：真实车辆 App 中的敏感算法和密钥并不一定存放在 Java 层，也可能隐藏在 Native SO 文件中。

## 七、PWN 方向赛题剖析

### 7.1 pwn-3

#### 1. 题目原理

pwn-3 是一道经典 Linux 栈溢出题。

程序在栈中定义了长度为 128 字节的缓冲区，但使用 `read()` 读取最多 `0x100` 字节数据，从而导致缓冲区溢出。

漏洞形式类似：

```c
char buf[128];
read(0, buf, 0x100);
```

攻击者可以通过超长输入覆盖：

- 保存的栈帧；
- 返回地址；
- 后续 ROP 链。

#### 2. 主要解题思路

典型利用过程为：

```text
检查程序保护
    ↓
确认栈溢出偏移
    ↓
寻找 ROP Gadget
    ↓
调用输出函数泄露 libc 地址
    ↓
计算 libc 基址
    ↓
计算 system 和 "/bin/sh" 地址
    ↓
构造 ret2libc
    ↓
获取 Shell
```

主要步骤包括：

1. 使用 `checksec` 检查保护机制；
2. 使用循环字符串或调试器确定覆盖返回地址的偏移；
3. 寻找 `pop rdi; ret` 等 Gadget；
4. 调用 `puts` 或 `write` 泄露 GOT 表中的 libc 函数地址；
5. 根据泄露地址计算 libc 基址；
6. 计算 `system` 和 `/bin/sh` 地址；
7. 构造第二阶段 ROP 链；
8. 获取远程 Shell；
9. 读取 Flag。

#### 3. 考查知识点

该题主要考查：

- Linux 二进制分析；
- 栈溢出；
- x86-64 调用约定；
- ROP；
- GOT 和 PLT；
- libc 地址泄露；
- ret2libc；
- pwntools。

#### 4. 汽车安全特征

pwn-3 本质上属于标准 PWN 题目，没有明显展示车载系统特征。

在真实汽车安全中，类似漏洞可能存在于：

- T-Box 网络服务；
- 车机 Linux 守护进程；
- SOME/IP 服务；
- DoIP 服务；
- 蓝牙服务；
- OTA 客户端；
- ECU 诊断程序；
- 多媒体解析组件。

如果题目运行于 ARM 或 MIPS 架构，并结合车载协议输入、T-Box 固件或车载 Linux 服务，其汽车安全属性会更明显。

## 九、第一届 CCF 线上赛与传统 CTF 的不同点

### 9.1 题目数据具有汽车业务语义

传统 CTF 中常见的数据包括普通用户信息、网站数据、随机密文和通用程序。

本次线上赛中出现了：

- VIN；
- GPX 轨迹；
- 车辆蓝牙通信数据；
- OTA 握手；
- 车辆型号；
- 电量；
- 里程；
- 车辆访问凭证；
- 车辆升级文件。

这些数据要求选手理解车辆业务字段的含义。

### 9.2 更强调跨层攻击链

传统 CTF 题目经常围绕单一漏洞或单一算法展开，而车联网题目更容易将多个技术方向组合起来。

例如 VIN 题的分析链路为：

```text
蓝牙通信流量
    ↓
Android APK
    ↓
JNI
    ↓
Native SO
    ↓
自定义密码算法
    ↓
VIN 数据
```

OTA-流量题则为：

```text
网络流量
    ↓
握手协议
    ↓
VIN 身份
    ↓
会话密钥
    ↓
加密升级包
    ↓
OTA 数据解密
```

这种跨层组合更接近真实车联网系统的安全分析过程。

### 9.3 业务字段可能成为攻击入口

在普通 CTF 中，字符串字段通常只是程序参数。

在车联网系统中，不同字段可能直接影响车辆权限和业务流程，例如：

- VIN 决定车辆身份；
- 车型决定升级包类型；
- 电量决定是否允许升级；
- 里程可能影响维护策略；
- Token 决定 OTA 会话权限；
- 特殊 VIN 可能触发管理员功能；
- 车辆访问凭证可能用于解锁或控制车辆。

因此，车联网安全不仅要寻找代码漏洞，也需要分析业务逻辑漏洞。

### 9.4 附件类型更加多样

本次比赛除传统 ELF、PHP、Python 文件外，还出现：

- APK；
- SO；
- PCAP；
- GPX；
- OTA 文件；
- 蓝牙通信记录；
- JSON 握手数据。

选手需要具备更强的多类型数据处理能力。
